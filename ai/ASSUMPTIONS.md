Assumptions and Questions for Frontend

Assumptions made while designing the schema:
1. Suggested vs Confirmed
   - suggested (boolean) indicates the item was generated by the AI pipeline.
   - confirmed (boolean) indicates the user explicitly accepted/edited the item.
   - Both fields can be true (AI suggested and user confirmed) or false (user created manually).

2. IDs and storage
   - id fields are stable UUIDs generated by backend.
   - Photos are stored in external object storage; the activity record keeps references (URLs).

3. Timestamps
   - All timestamps stored in ISO 8601 with timezone. Backend prefers UTC but accepts timezone-aware input.

4. Place linking
   - placeId is optional. When present, it should be a canonical external ID (Google Place ID, Foursquare, or internal place table id). placeName is stored redundantly for display.

5. Photos
   - photos array contains lightweight metadata only (no binary). Frontend fetches images by URL.

6. Confidence
   - confidence exists only for suggested items but is allowed on any item. Range: 0.0–1.0.

7. Metadata
   - metadata is freeform JSON for detector outputs. Keys should be short and predictable (e.g., detector, source, tags).

8. Mutability
   - Activities are mutable. updatedAt must change on edits. Keep an audit log externally if needed.

Questions for frontend / product:
1. UX for suggested items: should suggested items be visually distinct (e.g., dashed border) and require explicit confirmation to be considered "logged"?
2. Editing flow: when user edits a suggested item, do we set confirmed=true automatically or wait for an explicit "Confirm" action?
3. Photos: should adding/removing photos be separate confirmation actions or part of the single activity confirmation?
4. Place linking: should the frontend allow selecting a place from a list (with placeId) or only free-text placeName?
5. Offline/optimistic edits: can frontend create activities locally (without id) and reconcile after backend assigns id? If yes, please describe conflict resolution preferences.
6. Sorting: default client sort is by startTime ascending. Is that acceptable across views?
7. Privacy: any items with placeId or photos that are sensitive should have an opt-out flag — do we need an isPrivate boolean on the activity?

Recommended frontend contracts (quick):
- PATCH /activities/:id -> accepts partial fields, returns full activity with updatedAt.
- POST /activities -> create activity, return id and createdAt.
- GET /days/:date/activities -> return array of activities for that day ordered by startTime.
- POST /activities/:id/confirm -> mark confirmed=true (idempotent).

If you want, we can add an explicit isPrivate field to the schema; we left it out to avoid premature complexity but can add it quickly.
